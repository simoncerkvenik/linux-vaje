# 09.1 - Troubleshooting a degraded system state and NetworkManager behavior

## Purpose

This file contains a practical Linux troubleshooting example based on a real session with `systemd`, a failed unit, and `NetworkManager`.

The goal of this exercise was to:

- identify why the system state was `degraded`
- find the failed unit
- inspect its status
- apply a fix
- verify the result
- understand the difference between a service, a network interface, and active connectivity



## Initial symptom

I started with:

```bash
systemctl status
```

The output showed that the system state was:

- `State: degraded`
- `Failed: 1 units`

This meant that at least one systemd unit had failed.
# Step 1: Find the failed unit

To identify the problem, I used:

```bash
systemctl --failed
```

The failed unit was:

```bash
casper-md5check.service loaded failed failed Verify Live ISO checksums
```

This was the unit responsible for the degraded state.

---

## Step 2: Inspect the failed service

Next, I checked the service in more detail:

```bash
systemctl status casper-md5check.service
```

Important observations from the output:

- the service description was `Verify Live ISO checksums`
- the unit was in a `failed` state
- the logs showed messages related to checksum verification
- the output suggested that a required md5 file could not be found or opened

This suggested that the service was related to a live ISO verification process rather than a critical daily-use system service.

---

## Step 3: Disable the failed service

Because this service was not needed for normal installed-system operation, I disabled it:

```bash
sudo systemctl disable casper-md5check.service
```

This removed its automatic startup link.

---

## Step 4: Clear the failed state

After disabling the service, I cleared the stored failed state:

```bash
sudo systemctl reset-failed
```

Important note:

The command must be written exactly like this.

Correct:

```bash
sudo systemctl reset-failed
```

Incorrect examples that can cause errors:

```bash
sudo systemctl reset-failed systemctl
sudo systemctl reset-failed systemctl status
```

If extra words are added, systemd may interpret them as unit names

## Step 5: Verify the result

After resetting the failed state, I checked again:

```bash
systemctl --failed
systemctl status
```

The final result was:

- `Failed: 0 units`
- `State: running`

This confirmed that the degraded state had been resolved.

---

## Reading the final system status

After the fix, `systemctl status` showed a healthy state.

Important lines:

- `State: running`
- `Failed: 0 units`

This means:

- no failed units remain
- systemd considers the system healthy
- the previous degraded state is gone

The output also showed:

```text
Tainted: local-hwclock
```

This is not the same as a failed service.

It usually means the hardware clock is being treated as local time instead of UTC, which can happen in dual-boot setups or specific clock configurations.

It is a note, not the cause of the degraded state.
